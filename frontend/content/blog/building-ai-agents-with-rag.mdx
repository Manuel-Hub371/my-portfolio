Retrieval-Augmented Generation (RAG) has become the standard pattern for grounding large language models in private, up-to-date knowledge. In this article, I walk through a production-ready architecture.

## Why RAG?

Pure LLMs hallucinate on domain-specific facts. RAG injects relevant context at inference time, improving accuracy while keeping models smaller and cheaper.

## Architecture Overview

1. **Ingestion** — Parse PDFs, docs, and wikis into chunks with metadata.
2. **Embedding** — Store vectors in PostgreSQL with `pgvector` or a dedicated vector DB.
3. **Retrieval** — Hybrid search combining dense embeddings and BM25 sparse retrieval.
4. **Generation** — Prompt the LLM with retrieved chunks and strict citation rules.

```python
from langchain.chains import RetrievalQA
from langchain.vectorstores import PGVector

vectorstore = PGVector(connection_string=DATABASE_URL, embedding_function=embeddings)
qa = RetrievalQA.from_chain_type(llm=llm, retriever=vectorstore.as_retriever())
```

## Key Lessons

- Chunk size matters: 512–1024 tokens with overlap works well for technical docs.
- Always log retrieval scores for debugging bad answers.
- Use rerankers when precision is critical.
